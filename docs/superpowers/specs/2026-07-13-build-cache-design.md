# 构建与测试图片转换缓存设计

日期：2026-07-13
状态：设计已确认，待书面评审

## 1. 目标

保留当前无损 WebP 转换与像素一致性校验，同时消除测试和重复构建中对同一源图的重复转换。

验收目标：

- 首次无缓存构建仍允许约 13 秒；
- 后续测试与构建复用已经验证过的转换结果；
- 同一台机器上的第二次 `npm test` 目标不超过 5 秒；
- 热缓存下单个 validator 测试不再触发 Sharp 转换，目标不超过 1 秒；
- 页面内容、图片像素与最终 HTML 不发生变化。

## 2. 根因

当前 `tests/build.test.mjs` 调用 `buildPage()` 13 次，`tests/content.test.mjs` 再调用 1 次。每次构建都串行转换 10 张图片。

现场计时：

- 10 张图完整转换约 13.28 秒；
- 读取 13.8MB 的现有 HTML 约 7ms；
- 单次 HTML validator 平均约 87ms。

因此耗时来自重复执行“源图解码 → lossless WebP effort 6 → WebP 解码 → RGBA 像素比较”，不是 validator 或文件读取。

## 3. 方案比较

### 3.1 逐图片持久缓存（采用）

每张图片按内容和转换环境生成缓存键。命中时直接读取此前已完成像素校验的转换结果；只让实际变化的图片重新转换。

优点：

- 跨测试文件、Node 进程和构建命令复用；
- 修改一张图只失效一张图；
- 页面模板、CSS 和客户端脚本仍每次实时读取；
- 不需要依赖测试执行顺序。

### 3.2 整页 HTML 缓存（不采用）

整页命中速度最快，但模板或正文的任意小改动都会让 10 张图重新转换；缓存粒度过粗，也更容易隐藏页面输入遗漏。

### 3.3 仅测试内共享 fixture（不采用）

改动最小，但只能在一个测试进程内复用，无法覆盖 `content.test.mjs`、后续 `npm run build` 或下一次测试运行。

## 4. 缓存键与失效规则

缓存键使用 SHA-256，稳定包含：

- 源图片字节；
- 图片 manifest 中与结果有关的字段：`id`、`source`、`preset`、`webPMode`；
- `scripts/image-pipeline.mjs` 的文件内容；
- Sharp 与 libvips 的版本信息。

失效行为：

- 图片内容或 manifest 配置变化：只失效对应图片；
- 转换算法、质量参数、像素校验逻辑变化：全部图片自动失效；
- Sharp 或 libvips 版本变化：全部图片自动失效；
- 模板、CSS、capture bootstrap 或客户端脚本变化：这些文件不进入图片缓存，每次构建都会重新读取和组装，因此修改立即出现在产物中，不存在旧页面命中问题。

## 5. 缓存存储与数据流

缓存目录为 `.cache/image-pipeline/`，继续由现有 `.gitignore` 排除。

单个缓存文件保存一个已验证的转换结果，包括尺寸、字节数、MIME、像素一致性状态和 Data URI。流程：

1. 读取源图并计算缓存键；
2. 尝试读取对应缓存文件；
3. 命中且结构有效时返回结果，并标记 `cacheHit: true`；
4. 未命中或缓存损坏时执行现有完整转换与像素校验；
5. 校验通过后先写临时文件，再原子重命名为正式缓存；
6. 返回结果并标记 `cacheHit: false`。

损坏、截断或字段不完整的缓存不得导致构建失败或返回半成品，应删除或覆盖并重新转换。

`buildPage()` 仍然每次重新读取模板、CSS 和 JavaScript，替换占位符，清理 legacy HTML，并写出 `dist/index.html`。

## 6. 接口

`convertAsset` 保持现有调用兼容，并增加可测试的第三个可选参数：

```js
convertAsset(asset, root = process.cwd(), {
  cacheDir = resolve(root, '.cache/image-pipeline'),
} = {})
```

测试使用临时缓存目录，不污染真实项目缓存。构建继续使用默认目录。

## 7. 测试

新增或调整测试覆盖：

- 相同源字节、manifest、pipeline 和版本生成相同缓存键；
- 源字节、preset、pipeline 内容或 Sharp 版本任一变化都会改变键；
- 第一次调用产生 miss 和缓存文件，第二次调用产生 hit，结果内容一致；
- 损坏缓存会安全回退为重新转换；
- 缓存写入使用唯一临时文件并原子替换；
- `buildPage()` 多次调用仍会重新读取页面源文件并写出当前 HTML；
- 冷、热各运行一次 `npm test`，记录总耗时和各 validator 用例耗时；
- 完整 `npm run verify` 通过。

性能目标用于本机验收，不写成容易受 CI 机器波动影响的硬时间断言。

## 8. 范围

本次不改变 WebP 参数、像素质量标准、图片 manifest、页面文案或版式；不提交 `.cache` 内容；不引入额外 npm 依赖。
